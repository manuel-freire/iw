### Eventos DOM

* Funci√≥n JS *registrada* en un *nodo* (~= etiqueta HTML) DOM
   - el navegador la llama cuando corresponde, en funci√≥n de su *tipo*
* Tipos de evento
   - De *rat√≥n*: el usuario coloca el puntero sobre un elemento, \
    √≥ interacciona con √©l usando el rat√≥n
   - De *teclado*: el usuario presiona teclas (DOM L3) \
   - De *HTML*: el usuario carga la p√°gina o uno de sus recursos, \
    √≥ modifica formularios, o cambia el tama√±o o posici√≥n de la ventana
   - De *mutaci√≥n*: el contenido del DOM de la p√°gina cambia \
    (uso infrecuente salvo por librer√≠as)
   - De *interfaz de usuario*: responden al 'enfoque' \
   o desenfoque de nodos DOM, o su activaci√≥n (uso poco frecuente)

## Validaci√≥n

### D√≥nde validar

* En el cliente (= el navegador), porque los usuarios
    + pueden no saber qu√© espera tu formulario
    + pueden no entender c√≥mo lo espera, y equivocarse de formato
    + pueden despistarse, y olvidarse de introducir ciertos campos, o introducir informaci√≥n inconsistente.

* En el servidor (= la aplicaci√≥n web de backend), porque
    + los usuarios pueden intentar romper tu aplicaci√≥n
    + tu aplicaci√≥n puede actuar como una API para otras aplicaciones, y √©stas pueden estar mal programadas

### Qu√© hacer con los errores

* en el *cliente*, **informa** y **ayuda** al usuario a corregir lo que est√© mal
* en el *servidor*, 
    + no proceses informaci√≥n en la que no conf√≠as
    + usa c√≥digos de estatus HTTP para quejarte. 
    + Y, si piensas que
        - no es posible que esto suceda sin *malas intenciones*, **no** des informaci√≥n extra
        - crees que puede deberse a una confusi√≥n usando la API, **debes** informar algo m√°s

### Mala validaci√≥n

* Cosas a **no hacer** en validaci√≥n **cliente**
    + usar `alert` para avisar de errores
        - los `alert` son molestos, y no muestran d√≥nde tienes que hacer los cambios.
        - *mejor:* mensajes **junto a los controles** err√≥neos, donde haya que solucionarlo.
    + dar mensajes que se quejan de un problema, pero no dicen c√≥mo solucionarlo
        - a no ser que tu aplicaci√≥n consista en un juego de acertijos, \
        o que necesites que tu usuario se sienta culpable y no quiera usar tu aplicaci√≥n
        - *mejor:* mensajes **que expliquen c√≥mo solucionar el problema**

~~~{.js}
const mal   = 'identificador de objeto err√≥neo';
const mejor = 'este identificador debe corresponder a un objeto v√°lido;'
    + 'haz click en el enlace para ver todos los objetos disponibles';
// y todav√≠a mejor: usar un 'select' √≥ similar.
~~~

* Cosas a **no hacer** en validaci√≥n **servidor**
    + confiar en que la validaci√≥n cliente funciona.
        - la web es un sitio grande y salvaje. \

    + olvidarte de elegir c√≥digos de estado
